## 1. System Overview

### 1.1 Application Identity & Purpose

WriteSpace is a fully client-side, single-page blog application designed as a demonstrable MVP for a B2C content management platform. It serves as a complete, self-contained blogging experience where every piece of functionality — from user registration and authentication to content creation, role-based access control, and administrative oversight — executes entirely within the user's browser. There is no backend server, no REST API, no database engine, and no external authentication provider. The application's identity is defined by this radical simplicity: it is a blog platform that requires zero infrastructure beyond a static file host, yet delivers a complete, role-aware, multi-user experience that would traditionally demand a server stack of at least an application server, a database, and an authentication service.

The application addresses a specific and well-defined need: providing a fully functional, immediately deployable blogging platform for demonstrations, prototyping, and personal use where the overhead of server provisioning, database management, and API development is neither justified nor desired. WriteSpace proves that meaningful, role-based web applications can be built using only browser-native capabilities — the localStorage API for structured data persistence, the Web Crypto API for UUID generation, and React for reactive, component-based UI rendering. This is not a toy; it is a complete system with nine distinct routes, two user roles, ownership-based access control, an administrative dashboard, and a user management panel, all operating within the browser sandbox.

WriteSpace serves two distinct audiences whose needs and privileges differ fundamentally. The **Admin** is the site owner — a superuser with unrestricted access to all content and user accounts across the platform. The Admin can create, edit, and delete any blog post regardless of authorship, access a dedicated dashboard that surfaces platform-wide statistics (total posts, total users, admin count), and manage the full user roster through a dedicated administration panel that supports creating new accounts with either role and deleting existing ones. The **user** is a registered author who can create new blog posts, edit and delete their own content, and read all published posts across the platform. This two-role model establishes a clear, enforceable permission boundary that mirrors real-world content management systems — WordPress, Medium, Ghost — while remaining simple enough to implement entirely in client-side code without a server-side authorization service.

The business value of WriteSpace lies in its zero-cost operational model. Deployed as static files on Vercel's free tier, with all data residing in each user's browser localStorage, the platform incurs no hosting costs, no database provisioning fees, no API gateway charges, and no per-request pricing. This makes it an ideal vehicle for demonstrating content management concepts to stakeholders, prototyping role-based access patterns before committing to a backend architecture, or serving as a personal journaling tool where data privacy is guaranteed by the simple fact that data never leaves the user's machine. The key performance indicators that WriteSpace targets include user sign-up conversion (measured by the number of successful registrations stored in writespace_users), content creation volume (tracked via the growing writespace_posts array), and administrative efficiency (the speed with which an Admin can perform moderation tasks like deleting a post or creating a user account).

### 1.2 Execution Model: Browser-Only SPA

WriteSpace operates under a strict browser-only execution model that fundamentally shapes every architectural decision. When a user navigates to the deployed URL, Vercel's global CDN serves a single HTML document (`index.html`), a JavaScript bundle produced by Vite's production build, and an associated CSS stylesheet generated by Tailwind's JIT compiler. From that point forward, every interaction — navigation between routes, form submission and validation, data retrieval, data mutation, authentication credential checking, and authorization enforcement — occurs entirely within the browser process. There are no `fetch()` calls, no `XMLHttpRequest` invocations, and no WebSocket connections. The application is a closed system whose entire runtime state is bounded by the lifetime of the browser tab.

This execution model is made possible by the convergence of three complementary technologies, each addressing a distinct architectural concern. First, **React 18** provides the component model and reactive rendering engine that transforms application state into DOM updates. When a user submits a new blog post, React re-renders only the affected component subtree, not the entire page. Second, **React Router v6** provides client-side routing that intercepts URL changes — whether triggered by `<Link>` clicks, `useNavigate()` calls, or direct browser address-bar entry — and maps them to component trees without triggering full page reloads. This is the defining characteristic of a Single-Page Application: the browser never performs a full document navigation after the initial load. Third, the browser's **localStorage API** serves as the persistence layer, storing three distinct data collections as JSON-serialized strings: `writespace_posts` (an array of blog post objects, each with id, title, content, createdAt, authorId, and authorName), `writespace_users` (an array of registered user account objects, each with id, displayName, username, password, role, and createdAt), and `writespace_session` (a singleton object with userId, username, displayName, and role representing the currently authenticated user).

The application lifecycle follows a predictable, deterministic pattern that is easy to reason about. On initial load, the SPA bootstraps by calling `getSession()` — a utility function that reads `writespace_session` from localStorage and parses it from JSON. If a valid session object exists, the application renders the authenticated navigation shell (`<Navbar>`) and programmatically routes to the role-appropriate home page: `/admin` for users whose session role is `"admin"`, `/blogs` for users whose session role is `"user"`. If no session exists, the public navigation shell (`<PublicNavbar>`) is rendered and the landing page or authentication pages are displayed based on the current URL. Every subsequent data operation — listing all posts, creating a new user, editing blog content, deleting a post — reads from and writes to localStorage synchronously through the `storage.js` utility module. Because localStorage access is synchronous (the API blocks the main thread for the duration of the read/write, typically microseconds for the data volumes involved) and the data volumes are small (tens to hundreds of records, each a few kilobytes at most), all operations complete in well under a millisecond. This contributes directly to the near-instant perceived performance target of sub-100ms route transitions.

The Vite build toolchain serves as the bridge between development and production. During development, Vite provides sub-second Hot Module Replacement (HMR) that updates components in the browser without a full page reload. During the production build, Vite compiles React JSX source files into optimized JavaScript, resolves ES module imports into a bundled dependency graph, applies tree-shaking to eliminate unused code, and coordinates with Tailwind CSS's JIT compiler to generate only the utility classes actually used in the project. The output is a set of static assets — an HTML file, a hashed JavaScript bundle, and a hashed CSS file — that together constitute the complete, self-contained deployable artifact. The `vercel.json` configuration file contains a single SPA rewrite rule that instructs Vercel's edge servers to serve `index.html` for every URL path, allowing React Router to take over routing client-side. This ensures that direct navigation to `/blogs`, `/admin`, or `/blog/some-uuid` does not produce a 404 error — a critical requirement for any SPA deployed to a static host.

The enforcement architecture uses a layered defense model that addresses a critical security concern in client-side applications: the fact that any client-side check can be bypassed by a determined user with browser developer tools. The **outer layer** is the `<ProtectedRoute>` wrapper component. This component reads `writespace_session` from localStorage on every render. If no session exists, it renders a `<Navigate to="/login" />` element, which triggers React Router to redirect without adding a history entry. If the optional `role` prop is provided (e.g., `role="admin"`), the wrapper performs a second check: does `session.role` match the required role? If not, it redirects to `/blogs`. This outer layer handles the binary access decisions for six of the nine routes.

The **inner layer** consists of ownership checks embedded directly within the `WriteBlog` and `ReadBlog` components. These checks compare the post's `authorId` (retrieved from `writespace_posts` by matching the URL parameter `:id`) against the session's `userId`. This inner layer is essential because the `ProtectedRoute` wrapper only verifies that the user is authenticated — it does not know which specific post is being accessed and therefore cannot enforce ownership. A user could manually type `/edit/some-uuid` for a post they do not own, pass through the `ProtectedRoute` gate (they are authenticated), and reach the `WriteBlog` component. The in-component ownership check catches this violation on mount — before the form is rendered, before any data is pre-filled, and critically, before any save operation can occur — and redirects the user to `/blogs`. This dual-layer approach ensures that the UI's visual hiding of buttons (the "soft" enforcement that improves UX by not showing irrelevant controls) is backed by programmatic guards (the "hard" enforcement that prevents unauthorized mutations even if URL manipulation is attempted).

### 1.5 Role Model: Admin vs User

WriteSpace implements a two-role permission model that is deliberately simple yet sufficient to demonstrate real-world access control patterns including privilege separation, ownership-based authorization, and administrative oversight. The roles are stored as a string field (`role`) on each user object in the `writespace_users` array and on the session object in `writespace_session`. The possible values are strictly `"admin"` and `"user"` — there is no concept of a "super admin," "moderator," or "editor" role, keeping the permission matrix tractable for a 5–7 day MVP.

The **Admin role** represents the site owner — a superuser with unrestricted privileges across the entire platform. The Admin's capabilities span four dimensions. First, **content moderation**: the Admin can create, read, edit, and delete any blog post regardless of who authored it. On the blog list page (`/blogs`), the Admin sees a pencil Edit icon on every post card. On the read page (`/blog/:id`), the Admin sees both Edit and Delete buttons on every post. Second, **platform analytics**: the Admin has exclusive access to the `/admin` dashboard, which displays four computed statistics (total posts, total users, total admins, total regular users) derived by reading and aggregating data from `writespace_posts` and `writespace_users`. Third, **user administration**: the Admin has exclusive access to the `/users` management panel, where they can create new user accounts with either the Admin or user role, and delete existing accounts (with safeguards preventing deletion of the hard-coded admin account and self-deletion). Fourth, **visual distinction**: the Admin's avatar — rendered throughout the interface in the navbar chip, blog post author lines, and user management table — uses a crown emoji (👑) on a violet background (`bg-violet-600`), providing immediate visual identification of administrative presence.

The **user role** represents a registered author with ownership-scoped privileges. A user can create new blog posts via the `/write` route, edit their own posts via `/edit/:id` (enforced by the in-component ownership check), and delete their own posts from the read view (enforced by the same ownership check before the `window.confirm()` dialog and localStorage mutation). They can read all published content across the platform — the blog list and read views are unrestricted for authenticated users — but cannot modify content authored by others. On the blog list page, a user sees edit icons only on cards where the post's `authorId` matches their session's `userId`; on all other cards, the edit icon is absent. On the read page, a user sees Edit and Delete buttons only on their own posts; on other authors' posts, they see only a "Back to All Posts" navigation button. The user's avatar uses a book emoji (📖) on an indigo background (`bg-indigo-500`).

The **hard-coded Admin account** (`username: "admin"`, `password: "admin"`) occupies a special position in the authentication architecture. This account exists outside the `writespace_users` array — it is checked first in the login flow, before any localStorage user search occurs. This design ensures that the root Admin can always log in even if the `writespace_users` array is corrupted, empty, or accidentally deleted. The hard-coded admin cannot be deleted through the User Management panel: its Delete button is permanently disabled and displays a tooltip reading "Default admin cannot be deleted." Additionally, the currently logged-in Admin cannot delete their own account through the UI, preventing accidental self-lockout. These safeguards ensure that there is always at least one viable path to administrative access.

All **self-registered accounts** created via the `/register` page are unconditionally assigned the `"user"` role. There is no self-service path to Admin privileges — the registration form has no role selector, and the role is hard-coded to `"user"` in the registration logic. However, an Admin can create new Admin accounts through the User Management panel by selecting "Admin" from the role dropdown in the Create User form. This design ensures that the initial Admin (the hard-coded account) retains exclusive control over privilege escalation while allowing the creation of additional administrators for operational purposes such as shared site ownership.

### 1.6 Key Architectural Constraints

WriteSpace operates under a set of deliberately imposed constraints that define its architectural boundaries and distinguish it from a traditional web application. These constraints are not limitations to be overcome but rather the defining characteristics that make WriteSpace what it is — a zero-infrastructure, browser-only demonstration platform. Each constraint represents a conscious trade-off that was evaluated against alternatives and accepted because the benefits (simplicity, zero cost, rapid development) outweigh the costs for the MVP use case.

The most fundamental constraint is the **absence of a backend**. There is no application server, no REST API, no GraphQL endpoint, no WebSocket server, and no serverless function layer. Every computation — authentication credential verification, authorization decisions, data validation, CRUD operations, statistical aggregation — executes in the user's browser using JavaScript. This constraint eliminates entire categories of architectural concern: there is no network latency to budget for, no API versioning to manage, no CORS policies to configure, no rate limiting to implement, no database connection pooling to tune, and no server-side security vulnerabilities (SQL injection, SSRF, command injection) to mitigate. The trade-off is that data is siloed to a single browser — there is no multi-device synchronization, no collaborative editing, no data sharing between users on different machines, and no data recovery if localStorage is cleared or the browser's storage quota is exceeded. For a demo MVP, this trade-off is not only acceptable but desirable: it guarantees data privacy by construction (data never leaves the user's machine) and eliminates the operational burden of managing server infrastructure.

The second constraint is **plain-text password storage**. User passwords are stored as raw, unhashed strings in the `password` field of each user object in the `writespace_users` array. There is no hashing (bcrypt, SHA-256), no salting, and no encryption at rest. This constraint is explicitly documented in code comments throughout the `storage.js`, `auth.js`, `LoginPage.jsx`, and `RegisterPage.jsx` files with warnings that this practice is acceptable only for demonstration purposes and that no real personally identifiable information should be used during testing. This constraint is acceptable because WriteSpace is an MVP demo, not a production system handling real user data. In a production evolution of this platform, password hashing via the Web Crypto API's `SubtleCrypto` interface would be the first security upgrade, allowing passwords to be hashed client-side before storage while still avoiding the need for a backend.

The third constraint is the **exclusive use of JavaScript JSX**. No TypeScript files (`.ts` or `.tsx`) are permitted anywhere in the project. All React components use the `.jsx` extension; only pure utility files with no JSX syntax use `.js`. This constraint reduces toolchain complexity (no TypeScript compiler, no type definition management, no `tsconfig.json`), keeps the codebase accessible to developers who may not be familiar with TypeScript's type system, and aligns with the project's "no over-engineering" philosophy. The trade-off is the loss of compile-time type checking, which means certain categories of bugs — typo'd property names on objects read from localStorage, incorrect function argument types, missing required props on components — will only be caught at runtime. For a project of WriteSpace's scale (approximately 15 components, 2 utility modules), the surface area for such bugs is small enough that runtime testing provides adequate coverage.

The fourth constraint is the **exclusive use of Tailwind CSS utility classes** for all styling. No custom CSS files beyond `index.css` (which contains only the three `@tailwind` directives: `base`, `components`, `utilities`) are permitted. No CSS Modules, no styled-components, no inline `<style>` tags, and no separate stylesheets per component. This constraint enforces design-system consistency by limiting developers to Tailwind's curated set of design tokens (spacing scale, color palette, typography scale, breakpoint system). It prevents the style fragmentation that inevitably occurs when developers create ad-hoc stylesheets, and it keeps all visual decisions co-located with the markup they affect, making it easier to understand how a component looks by reading its JSX. The trade-off is that JSX markup becomes more verbose with long class strings (e.g., `bg-indigo-600 text-white px-4 py-2 rounded-lg font-medium shadow-sm hover:bg-indigo-700`), though this is mitigated by the component architecture which encapsulates repeated style patterns (buttons, cards, form fields) within reusable components.

The fifth constraint is the **prohibition of external state management libraries**. Only React's built-in `useState` and `useEffect` hooks may be used for state management. No Redux, no Zustand, no Jotai, no Recoil, and no React Context API for shared state. This constraint keeps the dependency footprint minimal (only `react`, `react-dom`, and `react-router-dom` as runtime dependencies) and forces a simple, localized state architecture where each component owns its state and derives shared data from direct localStorage reads. The trade-off is that there is no centralized state store, no time-travel debugging, no middleware for side effects, and no normalized data caching — but for an application of WriteSpace's scale, where the entire "database" is a synchronous key-value store readable in microseconds, these capabilities are unnecessary overhead that would add complexity without proportional value.

The sixth constraint is the **single-environment deployment model**. WriteSpace is deployed exclusively as a static site on Vercel's production environment. There is no staging environment, no testing environment, no preview deployments per pull request, and no development environment in the cloud. Local development uses Vite's dev server on `localhost:5173`. The `vercel.json` file contains only the SPA rewrite rule — no `buildCommand`, no `outputDirectory`, no `installCommand`, no `framework`, no `builds`, and no environment variables. This constraint reflects the project's nature as a demo MVP built by a single developer in 5–7 days: the complexity of a multi-environment CI/CD pipeline with branch-based preview deployments is not justified for a project of this scope and team size.

The final constraint is the **dependency on browser localStorage availability**. The entire application's data persistence depends on the `window.localStorage` API being present and functional in the user's browser. In private browsing modes (Safari's Private Browsing, Firefox's Private Window), some browsers historically restricted localStorage, and in extremely locked-down enterprise environments, it may be disabled by policy. The mitigation strategy is defensive: all read operations in `storage.js` are wrapped in `try/catch` blocks with a fallback to an empty array `[]`, and all views render appropriate empty or error states when data is unavailable. This ensures that even if localStorage is completely unavailable, the application degrades gracefully rather than crashing with an unhandled exception.

## 2. Architecture Style & Rationale

### Pattern Selection: Single-Page Application

WriteSpace adopts the Single-Page Application (SPA) architectural pattern as its foundational structural paradigm. This decision is not merely a technology preference — it is a direct consequence of the system's most binding constraint: the complete absence of any backend server, REST API, or external database. In a traditional web architecture, the choice between SPA, Multi-Page Application (MPA), or hybrid server-rendered approaches is shaped by factors such as SEO requirements, time-to-interactive budgets, and the complexity of server-side data fetching. For WriteSpace, the constraint set eliminates entire categories of architectural patterns from consideration, leaving the SPA as the only pattern capable of delivering a fully functional, multi-route application with zero server infrastructure.

At its core, the SPA pattern means that the entire application — all HTML, JavaScript, CSS, and routing logic — is delivered to the browser as a single static payload from the Vercel CDN. Once loaded, React Router v6 assumes full control over navigation, intercepting URL changes and rendering the appropriate page component without ever issuing a network request to a backend. This is the defining characteristic that makes the SPA pattern indispensable for WriteSpace: every route transition is a purely client-side operation. The browser's History API is manipulated by React Router to update the address bar, while React's reconciliation algorithm efficiently swaps component trees in the DOM. The result is a navigation experience that feels instantaneous — typically well under the 100ms perceived latency target — because there is no server round-trip, no API call, and no database query to wait on.

The SPA pattern also aligns with the Vite build toolchain in a natural and well-understood way. Vite produces a static bundle of ES modules, HTML, and assets that can be served from any static file host. The `vercel.json` rewrite rule — which maps all URL paths to `/index.html` — is the standard SPA deployment idiom: it ensures that when a user navigates directly to `/blogs` or refreshes the page on `/admin`, the static server returns the same `index.html` shell, and React Router picks up the URL to render the correct view. Without this SPA rewrite pattern, direct URL access would result in 404 errors because the static host has no file at `/blogs/index.html`. The SPA pattern, combined with the catch-all rewrite, solves this elegantly.

Within the SPA, WriteSpace implements a role-aware routing topology spanning nine distinct routes. The routing tree is organized around two orthogonal concerns: authentication state (public vs. protected) and role (admin vs. user). The `ProtectedRoute` wrapper component serves as a declarative guard, inspecting the `writespace_session` key in localStorage before allowing access to protected routes. This guard pattern is a well-established SPA idiom — it keeps authorization logic centralized and composable, avoiding scattered checks across page components. The ownership-based access control for edit and delete operations, however, is deliberately pushed down into the `WriteBlog` and `ReadBlog` components themselves, because ownership is a data-level concern that depends on the specific post being accessed, not merely on the route being visited.

The SPA pattern does introduce a well-known trade-off: the initial page load carries the entire application bundle. For WriteSpace, this is an acceptable cost because the application is intentionally small — a handful of page components, a few shared UI components, and two utility modules. The Vite-produced bundle, even without code splitting, is measured in tens of kilobytes, not megabytes. There is no large dependency tree, no heavy third-party libraries, and no media assets beyond the emoji-based avatar system. The time-to-interactive on a cold load over a typical broadband connection is well under one second, and subsequent navigations are effectively free. In a production-scale application with hundreds of routes and heavy dependencies, the SPA's bundle-size penalty would demand code splitting and lazy loading; for WriteSpace's scope, the simplicity of a single bundle outweighs the marginal performance gain of splitting.

### Client-Only Architecture Rationale

The decision to build WriteSpace as a client-only application — with no backend server, no API layer, no external authentication provider, and no cloud database — is the single most consequential architectural choice in the entire system. It is not a compromise born of resource constraints; it is a deliberate design position that maximizes the system's value along several dimensions that matter for an MVP demo: zero infrastructure cost, zero deployment complexity, complete data sovereignty for the user, and the ability to demonstrate a fully functional role-based application with nothing more than a static file host.

In a client-only architecture, the browser is the entire execution environment. There is no server-side rendering, no API gateway, no middleware pipeline, and no database connection pool. Every line of application logic — authentication, authorization, data validation, CRUD operations, session management — executes within the user's browser, using only the JavaScript runtime and the Web Storage API. This has profound implications for the system's trust model, which is explored in depth in the Security Architecture section, but at the architectural level it means that the application's runtime boundary is identical to the browser's sandbox. There is no server-side enforcement of any rule; every guard, every ownership check, and every validation is implemented in client-side JavaScript and is therefore inherently bypassable by a technically sophisticated user with access to the browser's developer tools.

Why accept this limitation? Because WriteSpace is explicitly scoped as a demonstration platform, not a production system handling real user data. The PRD and epic documentation repeatedly emphasize that no real personally identifiable information (PII) should be used, that passwords are stored in plain text with documented warnings, and that the system is suitable for prototyping, demos, and personal use. In this context, the absence of server-side enforcement is not a security vulnerability to be mitigated — it is an accepted architectural trade-off that eliminates an entire tier of infrastructure, dramatically reducing the system's operational surface area.

The client-only architecture also eliminates an entire category of failure modes. There is no server to crash, no database connection to time out, no API endpoint to return a 500 error, and no network partition to degrade functionality. The only failure domain is the browser itself: localStorage unavailability (handled via try/catch with empty-array fallbacks), JavaScript execution errors (contained within React error boundaries at the component level), and corrupted storage state (mitigated by defensive reads that never assume valid data). This radical simplification of the failure model is a significant architectural advantage for a demo application that must "just work" when shown to stakeholders.

From a deployment perspective, the client-only architecture reduces the operational footprint to a single static asset bundle served from a CDN edge network. Vercel's global CDN ensures that the initial HTML, JavaScript, and CSS payload is delivered from a point of presence geographically close to the user, minimizing cold-load latency. There is no database to provision, no environment variables to configure, no secrets to rotate, and no runtime to monitor. The entire deployment can be triggered by a single `git push` to a connected repository, and rollbacks are instantaneous because they involve nothing more than serving a previous version of static files.

The client-only pattern does impose one significant constraint on the development model: all data persistence must use browser-local storage APIs. There is no server-side database to fall back on, no cloud synchronization service, and no backup mechanism. Data lives and dies with the browser's localStorage. This constraint is addressed directly in the next subsection.

### LocalStorage as the Persistence Backend

The choice of the Web Storage API's `localStorage` as WriteSpace's sole persistence mechanism is a direct architectural consequence of the client-only pattern. With no backend server and no external database, the application must store all structured data — user accounts, blog posts, and session state — somewhere within the browser's sandbox. The browser platform offers several storage options: `localStorage`, `sessionStorage`, IndexedDB, the Cache API, and even in-memory JavaScript objects. Among these, `localStorage` was selected for three architectural reasons: its synchronous API simplifies the data access pattern, its string-based key-value model maps cleanly to JSON-serialized arrays and objects, and its 5–10 MB origin-level quota is more than sufficient for the demo-scale data volumes WriteSpace is expected to handle.

The synchronous nature of `localStorage` is particularly important in the context of React's rendering model. React's `useState` and `useEffect` hooks can read from `localStorage` directly during component initialization without needing to manage asynchronous promises or loading states. A utility function like `getPosts()` can synchronously parse `writespace_posts` from localStorage and return either the parsed array or an empty fallback — all within a single `try/catch` block. This eliminates an entire category of UI complexity: there are no loading spinners for data fetches, no skeleton screens, and no error states for failed network requests. The data is either present and valid, or it is absent and the UI renders an empty state. This binary simplicity is architecturally elegant for an application of WriteSpace's scope.

WriteSpace organizes its localStorage footprint into exactly three keys, each with a well-defined schema and lifecycle:

- **`writespace_posts`** — a JSON array of post objects, each containing an `id` (UUID), `title`, `content`, `createdAt` timestamp, `authorId`, and `authorName`. This array is the system's entire content repository. It grows monotonically as users create posts and shrinks only when posts are deleted. There is no pagination, no indexing, and no querying beyond what JavaScript's `Array.prototype` methods provide. For a demo with dozens or even hundreds of posts, linear scans are perfectly performant.

- **`writespace_users`** — a JSON array of user account objects, each containing an `id` (UUID), `displayName`, `username`, `password` (plain text, documented), `role` (either `"admin"` or `"user"`), and `createdAt` timestamp. This array serves as the system's user directory. The hard-coded admin account (`admin`/`admin`) exists outside this array — it is checked first in the login flow as a special case, ensuring the root admin account can never be accidentally deleted through the user management panel.

- **`writespace_session`** — a single JSON object (not an array) containing `userId`, `username`, `displayName`, and `role`. This key represents the currently authenticated user's session. It is written on successful login or registration, read by `ProtectedRoute` and navbar components to determine the current user's identity and role, and cleared on logout. There is no session expiry, no token refresh, and no multi-session support — the session is a simple, persistent marker of the last successful authentication.

Every read operation from localStorage is wrapped in a `try/catch` block with a fallback to an empty array (`[]`) for the collection keys or `null` for the session key. This defensive pattern is not optional — it is a hard architectural requirement driven by the reality that `localStorage` can throw exceptions in several scenarios: when the user's browser is in private browsing mode with storage disabled, when the storage quota is exceeded, or when the stored data has been corrupted by manual editing or a bug in a previous version of the application. The fallback-to-empty pattern ensures that the application degrades gracefully to a blank-slate state rather than crashing with an unhandled exception. This directly addresses NFR-003 and NFR-004.

The primary architectural limitation of localStorage is its lack of cross-device or cross-browser persistence. A user who registers on one browser will not see their account or posts on another device. This is an accepted limitation for the MVP scope — WriteSpace is a single-browser, single-user-at-a-time demonstration, not a multi-tenant collaborative platform. The PRD explicitly excludes concurrent user access and data synchronization from scope.

### Rejected Alternatives

Every architectural decision is made in the context of alternatives considered and rejected. For WriteSpace, several plausible architectural patterns were evaluated and consciously set aside. Documenting these rejections is essential for understanding why the chosen architecture is the right fit for this specific system at this specific point in its lifecycle.

**Server-Rendered Architecture (Next.js SSR/SSG):** A server-rendered React application using Next.js would provide server-side rendering for improved SEO and faster time-to-first-contentful-paint, along with API routes for backend logic. This pattern was rejected for two reasons. First, it would require a Node.js server runtime, violating the constraint of static-site-only deployment on Vercel without serverless functions. Second, it would introduce a backend tier — even if only Vercel serverless functions — which would need to manage database connections, environment variables, and deployment configuration, adding complexity disproportionate to the application's needs. WriteSpace does not need SEO (it is a demo, not a public-facing production site), and its bundle is small enough that client-side rendering is perceptually instantaneous.

**Backend-for-Frontend (BFF) Pattern:** A dedicated BFF layer — a lightweight server that sits between the SPA and a database, handling authentication, authorization, and data validation — would address the client-side trust problem by moving security-critical logic to a server environment. This pattern was rejected because it would require a persistent server process, a database (even SQLite would need a server or a WASM-based solution), and a deployment topology more complex than static hosting. The BFF pattern is the right choice for a production system handling real user data; for WriteSpace's MVP demo scope, the added complexity is not justified by the marginal security improvement.

**External Authentication Providers (OAuth, Firebase Auth):** Integrating a third-party authentication service like Firebase Authentication or Auth0 would eliminate the plain-text password storage concern and provide a more realistic auth flow. This was rejected because it would introduce an external dependency, require API keys and configuration, and violate the "no external API calls" constraint. The hard-coded admin and localStorage-based user registry, while architecturally simplistic, keep the system entirely self-contained.

**IndexedDB for Persistence:** IndexedDB offers a more capable storage API than localStorage — it supports structured data, indexes, transactions, and significantly larger storage quotas. It was rejected because its asynchronous API would require every data access in the application to be wrapped in promise-based patterns, adding complexity to React components that currently benefit from synchronous localStorage reads. For the data volumes WriteSpace handles (tens to hundreds of records, not thousands or millions), IndexedDB's performance advantages are irrelevant, and its API complexity is a net negative.

**State Management Libraries (Redux, Zustand, Context API):** The PRD explicitly prohibits external state management libraries, but the architectural rationale goes deeper than a simple constraint. WriteSpace's state is fundamentally derived from localStorage — the source of truth is the browser's storage, not an in-memory store. Components read from localStorage on mount (via `useEffect`) and write to localStorage on mutation. There is no complex cross-component state synchronization, no optimistic updates, and no cache invalidation. In this architecture, a state management library would be an abstraction layer over an abstraction layer — adding indirection without adding value. React's built-in `useState` and `useEffect` hooks are the right level of abstraction for this data flow pattern.

**TypeScript:** The decision to use plain JavaScript JSX rather than TypeScript is a project constraint, but it has architectural implications. TypeScript would provide compile-time type checking for the post, user, and session data structures, catching shape mismatches before runtime. It was rejected to keep the project accessible to developers who may not be familiar with TypeScript and to reduce the build toolchain complexity. The trade-off is that data shape errors — such as accessing a missing property on a post object — will only surface at runtime, requiring more thorough manual testing of edge cases.

### Trade-off Analysis & Risk Acceptance

Every architecture embodies a set of trade-offs — choices where a benefit in one dimension comes at a cost in another. The WriteSpace architecture makes several significant trade-offs that must be explicitly acknowledged, analyzed, and accepted before development proceeds. These are not oversights or gaps; they are conscious decisions made within the constraints of the MVP demo scope.

**Security vs. Simplicity:** The most significant trade-off is the acceptance of client-side-only security enforcement. In WriteSpace, route guards, role checks, and ownership validation all execute in the browser. A user with basic JavaScript knowledge can open the browser's developer console, modify `writespace_session` to set `role: "admin"`, and gain access to the admin dashboard and user management panel. They can directly manipulate `writespace_posts` to edit or delete any post. This is not a bug — it is an inherent property of a client-only architecture. The mitigation is not technical but contextual: WriteSpace is a demo application. It stores no real PII, uses no real passwords, and is not deployed in a context where unauthorized access would cause harm. The plain-text password storage in localStorage is documented with explicit warnings in code comments (per NFR-007), and the PRD instructs developers and testers never to use real credentials. This trade-off is acceptable for the MVP but would be unacceptable for any system handling real user data — at which point the architecture would need to evolve toward a BFF or server-rendered pattern.

**Data Durability vs. Zero Infrastructure:** localStorage provides no backup, no replication, and no recovery mechanism. If a user clears their browser data, switches to a different browser, or uninstalls the browser, all posts, user accounts, and session data are permanently lost. There is no "undo" for a cleared localStorage. This trade-off is accepted because the alternative — implementing a backup mechanism — would require either a server-side component (violating the client-only constraint) or a manual export/import feature (adding scope not requested in the PRD). For a demo application where data is expected to be ephemeral and recreated as needed, this is a reasonable position.

**Single-User Focus vs. Multi-User Reality:** WriteSpace implements a role-based permission model with Admin and user roles, but it does not support true multi-user concurrency. Two users cannot be logged in simultaneously on the same browser, because there is only one `writespace_session` key. If User A logs out and User B logs in on the same machine, User B sees all of User A's posts (because `writespace_posts` is shared) but cannot edit or delete them (because ownership checks compare `authorId` against the current session). This is architecturally correct behavior — the ownership model works as designed — but it highlights that localStorage is fundamentally a single-user persistence layer being used to simulate a multi-user system. This trade-off is accepted because the demo's purpose is to showcase the role and ownership model, not to serve as a production multi-user platform.

**No Offline/Online Distinction:** Because there is no network communication, WriteSpace has no concept of "offline" versus "online." The application is always available as long as the browser is running, but this also means there is no synchronization mechanism if the application were ever extended to include a backend. This is not a trade-off for the current scope — it is a simplification — but it becomes a migration challenge if the system ever evolves toward a connected architecture.

**Testing Surface Area:** The client-only architecture concentrates all logic in the browser, which means that end-to-end testing must account for localStorage state, browser routing behavior, and React component lifecycles. There is no API layer to test in isolation, no database to seed with fixtures, and no server middleware to unit-test. While this simplifies the testing toolchain (no need for API mocking libraries or database test containers), it shifts the testing burden toward integration and end-to-end tests that exercise complete user journeys. The architecture accepts this shift, relying on the small number of routes and well-defined user flows to keep the testing surface manageable.

In summary, the WriteSpace architecture — SPA pattern, client-only execution, localStorage persistence — is not a general-purpose blueprint for web applications. It is a purpose-built architecture optimized for a specific context: a zero-infrastructure, instantly deployable, role-aware blog demo that can be built by a single developer in under two weeks. The trade-offs it makes are severe by production standards but entirely appropriate for its intended use. The architecture document that follows describes how this pattern is realized in concrete modules, components, data flows, and deployment configurations.

### 3.2 Presentation Layer: Pages

The Presentation Layer comprises eight page-level components, each mapped to exactly one route (with the exception of `WriteBlog`, which serves two routes — `/write` and `/edit/:id` — by inspecting the URL to determine its mode). These pages are the largest modules in the system by responsibility surface area: each one owns a complete user journey, from data retrieval through rendering to mutation and navigation. They are not thin wrappers around shared components; they contain the imperative logic that stitches the application together — form validation, ownership enforcement, session inspection, and navigation decisions.

**LandingPage** is the only page accessible without authentication. It reads `writespace_posts` via `storage.js` to populate the latest-posts preview (up to three entries, sorted by `createdAt` descending), but it performs no writes. Its interaction with the session is read-only: it checks `auth.js` to decide whether the CTAs should point to `/login`/`/register` (guest) or to the role-appropriate dashboard (authenticated). The failure domain is narrow — if localStorage is corrupted, the latest-posts section gracefully degrades to the "No posts yet — check back soon!" empty state, and the hero and features sections render unaffected. This page depends on `PublicNavbar`, `BlogCard`, and the two utility modules.

**LoginPage** and **RegisterPage** form the authentication gateway. Both share a common architectural pattern: they render a centered white card on the full-viewport gradient background, they check for an existing session on mount (redirecting authenticated users away to avoid redundant login), and they own the complete form-submission lifecycle. LoginPage executes a two-tier credential check — hard-coded admin first, then a linear scan of `writespace_users` — and writes the session object via `auth.js` on success. RegisterPage performs four validation checks (all fields required, password match, username uniqueness against both the users array and the hard-coded admin string) before persisting the new user via `storage.js` and writing the session. Both pages depend on `storage.js` and `auth.js`; neither depends on any shared component beyond what `App.jsx` provides in the layout shell. Their failure domain includes duplicate-username races (two registrations in rapid succession), which are mitigated by performing the uniqueness check immediately before the write, though true atomicity is impossible in a synchronous localStorage model.

**Home** is the authenticated landing page for regular users and the blog-browsing surface for all authenticated roles. It reads the full `writespace_posts` array, sorts it by `createdAt` descending, and maps each post to a `BlogCard` component, passing the post data, the current session (for ownership determination), and the card's index (for accent-border cycling). Home performs no writes and owns no mutation logic — its responsibility ends at rendering the grid and the empty-state CTA. It depends on `storage.js`, `auth.js`, `Navbar`, and `BlogCard`.

**ReadBlog** is the most security-sensitive page in the system because it is the only page where both read and delete operations coexist behind an ownership gate. It resolves the post by ID from `writespace_posts` via `storage.js`, handles the not-found case with a dedicated error state, and then branches its control surface based on the intersection of the viewer's role and the post's `authorId`. The ownership check is performed twice — once to decide which buttons to render (UI-level gating) and once inside the delete handler before removing the post from the array (logic-level gating). This defense-in-depth approach ensures that a user who manipulates the DOM to reveal a hidden Delete button still cannot delete another author's post, because the delete handler independently verifies `authorId === session.userId || session.role === 'admin'` before calling `storage.savePosts()`. The delete workflow includes a `window.confirm` dialog, and on confirmation the post is filtered out of the array and the updated array is persisted before navigating to `/blogs`. ReadBlog depends on `storage.js`, `auth.js`, `Navbar`, and `Avatar`.

**WriteBlog** is the most complex page module because it serves two distinct modes — create and edit — through a single component that branches on the presence of the `:id` URL parameter. In create mode (`/write`), it initializes empty form fields, generates a UUID via `crypto.randomUUID()`, stamps the current timestamp and session-derived author fields, and appends the new post to `writespace_posts`. In edit mode (`/edit/:id`), it first resolves the existing post, then performs the ownership gate: if the current user is a regular user and the post's `authorId` does not match the session `userId`, the component immediately redirects to `/blogs` via `useNavigate`. If the gate passes (or the user is Admin), the form is pre-filled with the existing title and content. On save in edit mode, the post is updated in-place within the array (found by ID) and the array is re-persisted. Both modes share the same validation logic — both fields required, with inline error messages — and the same character counter below the content textarea. The Cancel button navigates back without persisting. WriteBlog depends on `storage.js`, `auth.js`, and `Navbar`.

**AdminDashboard** is the first page an Admin sees after login. It computes four aggregate statistics by reading `writespace_posts` and `writespace_users` and applying simple counts — total posts, total users, total admins (users with `role === 'admin'`), and total regular users (users with `role === 'user'`). These statistics are passed to `StatCard` components for rendering. The Recent Posts section reads the five most recent posts and renders them with inline Edit and Delete controls that navigate to `/edit/:id` or trigger the delete workflow directly. AdminDashboard depends on `storage.js`, `auth.js`, `Navbar`, and `StatCard`.

**UserManagement** is the admin-exclusive page for account lifecycle operations. It renders a create-user form at the top — with fields for display name, username, password, and a role dropdown (Admin/user) — and a user list below. The create flow validates all fields, checks username uniqueness (against both the users array and the hard-coded admin), generates a UUID and timestamp, and appends the new user to `writespace_users`. The delete flow includes a `window.confirm` dialog and two safety gates: the hard-coded admin account's delete button is permanently disabled with a tooltip, and the currently logged-in user cannot delete their own account (preventing accidental self-lockout). The user list is rendered responsively — as a table on desktop and as stacked cards on mobile — via the `UserRow` component. UserManagement depends on `storage.js`, `auth.js`, `Navbar`, `UserRow`, and `Avatar`.

### 3.3 Shared Component Layer

The Shared Component Layer contains seven reusable UI primitives that are consumed by one or more pages. These components are deliberately stateless with respect to application data — they receive everything they need via props and render deterministically. They never read from localStorage directly and never trigger navigation. This purity makes them trivially testable and safe to reuse across pages with different authorization contexts.

**Avatar** is the simplest module in the entire system. It exports a single pure function that maps a role string (`'admin'` or `'user'`) to a styled `<span>` element containing an emoji and Tailwind background classes. It has zero dependencies, zero props beyond the role string, and zero failure modes — it is a pure rendering function. Its placement in the shared layer, rather than being inlined in every consumer, ensures visual consistency: every avatar in the application — navbar chip, blog author line, user management table — is rendered by the same function, so a design change to the avatar style requires a single edit.

**StatCard** is a presentational tile for the admin dashboard. It receives a numeric value, a label string, and a color token, and renders a card with a colored icon background and the statistic. It has no knowledge of where the data comes from — the parent page computes the statistic and passes it down. This separation allows AdminDashboard to own the computation logic while StatCard owns only the visual presentation.

**BlogCard** is the most behavior-rich shared component. It receives a post object, the current session (or null, for guest contexts), and a zero-based index. From these inputs it derives: the excerpt (first 120 characters of content), the formatted date, the accent border color (via `index % 4` cycling through indigo, violet, pink, teal), and the visibility of the edit icon (always visible for Admin; visible for user only when `authorId === session.userId`; never visible for guests). The edit icon, when clicked, navigates to `/edit/:id`. The card body, when clicked, navigates to `/blog/:id` — or to `/login` if the viewer is a guest, implementing the gated preview behavior required by the landing page. BlogCard depends on `Avatar` for rendering the author avatar.

**UserRow** renders a single user record in two responsive variants: a table row (`<tr>`) on desktop and a stacked card on mobile, controlled by Tailwind's responsive utility classes (`hidden md:table-row` / `block md:hidden`). It receives the user object, a flag indicating whether this row represents the hard-coded admin, and a flag indicating whether this row represents the currently logged-in user. From these it derives: the delete button's disabled state (disabled for hard-coded admin and for self), the tooltip text, and the role badge styling. The delete action is surfaced via an `onDelete` callback prop — UserRow never mutates data directly. It depends on `Avatar`.

**PublicNavbar** and **Navbar** are the two navigation shells. PublicNavbar is shown on the landing page and conditionally adapts: if no session exists, it renders Login and Get Started buttons; if a session exists (a returning user who has not logged out), it renders an avatar chip with a "Go to Dashboard" button that links to `/admin` or `/blogs` based on role. Navbar is shown on all authenticated pages and renders role-based navigation links (Admin sees All Blogs, Write, and Users; user sees All Blogs and Write), highlights the active link with an indigo pill background, and provides a logout dropdown on the avatar chip. The mobile hamburger toggle is managed by local `useState` within Navbar — no external library is used. Both navbars depend on `auth.js` for session reading; Navbar additionally depends on `Avatar`.

### 3.4 Utility & Persistence Layer

The Utility & Persistence Layer is the foundation on which every other module rests. It consists of two plain JavaScript modules — `storage.js` and `auth.js` — that encapsulate all interaction with the browser's localStorage API. These modules contain no JSX, no React hooks, and no DOM interaction; they are pure data-access utilities that could be extracted and used in any JavaScript environment that provides a `localStorage`-like interface.

**storage.js** is the persistence gateway for the application's two entity collections: posts and users. It exports four functions that together form a minimal CRUD interface over the `writespace_posts` and `writespace_users` localStorage keys. The critical architectural decision embedded in this module is the defensive-read pattern: every read operation is wrapped in a `try/catch` block that catches `JSON.parse` failures (corrupted data) and `localStorage` access violations (private browsing restrictions, quota exceeded), returning an empty array `[]` as the fallback. This single decision eliminates an entire class of application crashes — without it, any corruption in localStorage would propagate as an unhandled exception through the React component tree, potentially white-screening the entire application. The write operations (`savePosts` and `saveUsers`) perform `JSON.stringify` and `localStorage.setItem` without fallback, on the assumption that write failures (quota exceeded) are both rare and unrecoverable — the application cannot function if it cannot persist data, and a caught write error would leave the in-memory state out of sync with the persisted state, creating a more confusing failure mode than an uncaught exception.

**auth.js** is the session-management module. It reads and writes the `writespace_session` key, which holds a single object representing the currently authenticated user. The session object contains four fields — `userId`, `username`, `displayName`, and `role` — and serves as the application's single source of truth for the current user's identity. Every authorization decision in the system — route guards, ownership checks, role-based UI visibility — ultimately traces back to the session object read by this module. The `clearSession` function removes the key entirely, which is the logout mechanism. Because the session is stored in localStorage (which survives browser restarts and tab closures), users remain logged in across sessions — a deliberate UX choice for a demo MVP that avoids the complexity of token expiration or refresh logic. The trade-off is that session invalidation is entirely manual (the user must explicitly log out), and there is no server-side session to revoke.

Both modules are imported by nearly every other module in the system. This creates a fan-out dependency pattern: a change to the localStorage key names or the session object shape would ripple through the entire application. To mitigate this risk, the key names (`writespace_posts`, `writespace_users`, `writespace_session`) are defined as constants within these modules and are never hard-coded in consumer modules — consumers call `getPosts()`, not `localStorage.getItem('writespace_posts')`. This encapsulation means that a key-name change requires edits only in `storage.js` and `auth.js`, not in the dozen-plus modules that depend on them.

### 3.5 Routing & Guard Layer

The Routing & Guard Layer is the architectural glue that maps URL paths to page components and enforces the application's access control matrix. It consists of two modules: `App.jsx`, the root orchestrator, and `ProtectedRoute.jsx`, the guard wrapper.

**App.jsx** is the application's entry point after `main.jsx` mounts the React tree inside `BrowserRouter`. It owns three responsibilities. First, it defines the complete route table — nine `<Route>` entries mapping paths to page components — using React Router v6's `<Routes>` and `<Route>` declarative API. Second, it wraps protected routes in `<ProtectedRoute>` (and, for admin-only routes, `<ProtectedRoute role="admin">`) to inject the guard logic without duplicating it across pages. Third, it conditionally renders the appropriate navbar — `PublicNavbar` for the landing page (`/`) and `Navbar` for all authenticated routes — by reading the session via `auth.js` and checking whether the current path is the public root. This conditional navbar rendering is placed in `App.jsx` rather than in a layout component because the navbar strategy is fundamentally a routing concern: which navbar appears is determined by which route is active and whether a session exists.
